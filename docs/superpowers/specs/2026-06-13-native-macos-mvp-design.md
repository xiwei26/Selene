# Native macOS MVP Design

## Summary

Build a separate native macOS client for Selene under `native-macos/`. The first release is a focused SwiftUI MVP that talks only to an existing Selene/MoonTV-compatible server. It does not replace or modify the current Flutter app, and it does not implement local subscription mode in the MVP.

The MVP success path is:

1. User launches the native macOS app.
2. User logs in with server URL, username, and password.
3. App stores the server URL and session cookie.
4. User searches through the configured server.
5. App shows search results.
6. User opens a result, chooses an episode, and plays it with a native macOS video player.

## Goals

- Create an independent pure-native macOS app that can be developed and shipped separately from the Flutter app.
- Preserve the existing Flutter multi-platform app and build paths.
- Use the current server APIs for login, search, detail, search resources, and playback URLs.
- Keep the MVP small enough to implement and verify quickly.
- Leave a clear extension point for local subscription mode without implementing it now.

## Non-Goals

- Replacing the existing `macos/` Flutter Runner.
- Rewriting Android, iOS, Windows, Linux, or web clients.
- Implementing local subscription mode.
- Implementing favorites, watch history, DLNA casting, live TV, recommendations, Douban details, Bangumi views, auto-update, or account management.
- Matching every visual detail of the Flutter app.
- Integrating mpv/libmpv in the MVP.

## Project Shape

Create a new folder:

```text
native-macos/
```

This folder contains a standalone SwiftPM macOS app rather than an Xcode project generated by Flutter. Keeping it separate avoids breaking `flutter build macos` and makes the native app easier to reason about.

Expected structure:

```text
native-macos/
  Package.swift
  Sources/SeleneNative/
    App/
      SeleneNativeApp.swift
    Views/
      RootView.swift
      LoginView.swift
      MainView.swift
      SearchResultsView.swift
      DetailView.swift
      PlayerView.swift
    Models/
      LoginSession.swift
      SearchResource.swift
      SearchResult.swift
    Services/
      ContentProvider.swift
      ServerAPIClient.swift
    Stores/
      SessionStore.swift
      SearchStore.swift
      PlayerStore.swift
    Playback/
      NativeVideoPlayer.swift
    Support/
      APIError.swift
      URLNormalizer.swift
  Tests/SeleneNativeTests/
```

The exact file names can be adjusted during implementation if SwiftPM or SwiftUI conventions require it, but the boundaries should remain stable.

## Architecture

The MVP uses SwiftUI for the interface, Foundation `URLSession` for networking, `UserDefaults` for lightweight session persistence, and AVKit for playback.

`ContentProvider` is the app-facing data interface. The MVP implementation is `ServerAPIClient`, which talks to the configured server. A future `LocalSubscriptionProvider` can implement the same interface later, but no local subscription UI or behavior is included in the MVP.

High-level flow:

```text
SwiftUI Views
  -> Stores
    -> ContentProvider
      -> ServerAPIClient
        -> Selene/MoonTV server APIs
```

Playback flow:

```text
SearchResult.episodes[index]
  -> PlayerStore
    -> NativeVideoPlayer
      -> AVPlayer/AVKit
```

## Server APIs

The native app should mirror the existing Flutter service behavior where relevant.

### Login

Request:

```http
POST {baseURL}/api/login
Content-Type: application/json
```

Body:

```json
{
  "username": "user",
  "password": "password"
}
```

Success:

- Status `200`
- Read `Set-Cookie`
- Store the cookie string for future requests

Failure:

- Show a concise login error.
- Do not enter the main app.

### Search

Request:

```http
GET {baseURL}/api/search?q={query}
Cookie: {storedCookie}
Accept: application/json
```

Expected response shape:

```json
{
  "results": [
    {
      "id": "string",
      "title": "string",
      "poster": "string",
      "episodes": ["https://..."],
      "episodes_titles": ["第1集"],
      "source": "string",
      "source_name": "string",
      "class": "string",
      "year": "string",
      "desc": "string",
      "type_name": "string",
      "douban_id": 123
    }
  ]
}
```

### Detail

Request:

```http
GET {baseURL}/api/detail?source={source}&id={id}
Cookie: {storedCookie}
Accept: application/json
```

Expected response shape is a single `SearchResult` object.

The app should call detail when a selected search result has missing or incomplete episode data, or when the user opens a result and a fresh detail request is desired.

### Search Resources

Request:

```http
GET {baseURL}/api/search/resources
Cookie: {storedCookie}
Accept: application/json
```

The MVP can use this endpoint as a connection/configuration check after login, but it does not need a full search-source management UI.

## Models

### SearchResult

Swift model fields should map to the current Dart model:

- `id: String`
- `title: String`
- `poster: String`
- `episodes: [String]`
- `episodeTitles: [String]`, decoded from `episodes_titles`
- `source: String`
- `sourceName: String`, decoded from `source_name`
- `className: String?`, decoded from `class`
- `year: String`
- `description: String?`, decoded from `desc`
- `typeName: String?`, decoded from `type_name`
- `doubanID: Int?`, decoded from `douban_id`

The model should be resilient to missing fields by using empty strings or empty arrays where the Flutter app currently does.

### SearchResource

Fields:

- `key: String`
- `name: String`
- `api: String`
- `detail: String`
- `from: String`
- `disabled: Bool`

The MVP does not expose resource editing.

### LoginSession

Fields:

- `serverURL: URL`
- `username: String`
- `cookie: String`

Password storage is deliberately omitted from `LoginSession`. The MVP can remember server URL, username, and cookie. Password persistence should be added later with Keychain if auto-login by password is required.

## UI Design

The app should feel like a regular macOS utility/player app, not a direct mobile layout port.

### Login Window State

If no usable session exists, show `LoginView`.

Fields:

- Server URL
- Username
- Password

Actions:

- Log in
- Clear form

States:

- Idle
- Logging in
- Error with retry

### Main Window State

After login, show `MainView` with a desktop split layout:

- Top toolbar/search area
- Left or center search results list/grid
- Right detail panel for the selected result
- Bottom or detail-embedded native player area

The exact visual layout can be refined during implementation, but the MVP must make these actions obvious:

- Search
- Select a result
- Select an episode
- Play/pause through native player controls
- Log out

### Search Results

Results should show:

- Poster when available
- Title
- Source name
- Year
- Type/class
- Episode count

Empty, loading, and error states must be explicit.

### Detail and Episode Selection

Detail panel should show:

- Title
- Source name
- Year/type
- Description when available
- Episode list

Episode title fallback:

- Use `episodeTitles[index]` when present.
- Otherwise display `第{index + 1}集`.

### Player

Use AVKit in the MVP.

Expected behavior:

- Plays the selected episode URL.
- Replaces the current item when the user selects another episode.
- Shows native playback controls.
- Surfaces a playback failure state when AVKit cannot load a URL.

Headers and advanced playback compatibility are not guaranteed in the MVP. If server-provided playback URLs require special request headers or codecs not supported by AVKit, that becomes an explicit post-MVP reason to evaluate mpv/libmpv.

## State Management

Use small `@Observable` stores on macOS 14+ where possible:

- `SessionStore`
  - Owns login state, persisted session, and logout.
- `SearchStore`
  - Owns query, loading state, results, selected result, and search errors.
- `PlayerStore`
  - Owns selected episode index, playback URL, and playback error.

Views should receive stores explicitly rather than relying on broad global state.

## Persistence

MVP persistence:

- Store server URL, username, and cookie in `UserDefaults`.
- Do not store password.

Future persistence:

- Add Keychain for password or token storage if auto-login by credentials is required.
- Add a local database or JSON storage if favorites/history are migrated.

## Error Handling

Errors should be user-readable and scoped to the failing surface.

Required cases:

- Invalid server URL.
- Login rejected.
- Network timeout or connection failure.
- Session expired or unauthorized response.
- Search returned no results.
- Detail request failed.
- Selected result has no playable episodes.
- AVKit playback failed.

For HTTP `401`, clear the stored cookie and return to the login state.

## Testing

Unit tests should cover:

- URL normalization for server URL input.
- Cookie extraction from login responses.
- `SearchResult` decoding with missing optional fields.
- `SearchResult` decoding of snake_case fields.
- API client request construction for login, search, and detail.
- Store state transitions for login success/failure and search success/failure.

Manual verification should cover:

- Launch app.
- Log in to a real server.
- Search a known title.
- Open a result.
- Select an episode.
- Confirm playback starts or a clear playback error is shown.
- Log out and confirm login view returns.

## Build and Run

Implementation should provide:

- `native-macos/script/build_and_run.sh`
- `native-macos/.codex/environments/environment.toml`

The script should build the SwiftPM app, stage a local `.app` bundle, launch it, and support a verification flag that confirms the app process exists.

## Future Work

Planned after MVP:

- Local subscription mode via `LocalSubscriptionProvider`.
- Favorites.
- Watch history and resume position.
- Better video engine with mpv/libmpv if AVKit compatibility is insufficient.
- DLNA casting.
- Live TV.
- Recommendations, Douban, and Bangumi surfaces.
- Keychain-backed credential storage.
- Native auto-update or release packaging.

## Open Decisions

These are intentionally deferred until implementation or post-MVP:

- Whether the main content uses a list-first or poster-grid-first result presentation.
- Whether playback is embedded in the detail area or opens in a dedicated player pane/window.
- Whether password auto-login is worth implementing immediately with Keychain.
- Whether AVKit compatibility is good enough for common server-provided episode URLs.
